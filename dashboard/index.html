Desain Arsitektur Data & Skema Database

Untuk sinkronisasi log sensor IoT (volume tinggi, time-series) dan riwayat pembayaran QRIS (transaksional, konsistensi tinggi), kami merekomendasikan pendekatan **Hybrid/Polyglot Database**:

1. Relational Database (e.g. PostgreSQL)

Digunakan untuk data keuangan, profil pengguna, dan audit transaksi yang memerlukan relasi kuat dan kepatuhan ACID.

-- Tabel: devices (Informasi fisik lot parkir)
CREATE TABLE devices (
    lock_id VARCHAR(50) PRIMARY KEY,
    location_name VARCHAR(100) NOT NULL,
    current_status VARCHAR(20) DEFAULT 'vacant', -- vacant, occupied, transition, error
    battery_percent INT CHECK (battery_percent BETWEEN 0 AND 100),
    solar_charging BOOLEAN DEFAULT FALSE,
    standby_current_ua INT DEFAULT 200, -- standby draw 200-300uA
    signal_strength INT,
    last_ping TIMESTAMP WITH TIME ZONE
);

-- Tabel: transactions (Data utama QRIS & sesi parkir)
CREATE TABLE transactions (
    transaction_id VARCHAR(100) PRIMARY KEY,
    lock_id VARCHAR(50) REFERENCES devices(lock_id),
    plate_number VARCHAR(15) NOT NULL,
    entry_time TIMESTAMP WITH TIME ZONE NOT NULL,
    exit_time TIMESTAMP WITH TIME ZONE,
    duration_minutes INT,
    amount_idr NUMERIC(10, 2) NOT NULL,
    qris_string TEXT NOT NULL,
    pg_reference VARCHAR(100),
    status VARCHAR(20) DEFAULT 'qris_generated', -- qris_generated, paid, expired, completed
    barrier_crosscheck VARCHAR(20) DEFAULT 'pending', -- pending, verified, manual_override
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

-- Tabel: manual_overrides (Log audit keamanan override)
CREATE TABLE manual_overrides (
    override_id SERIAL PRIMARY KEY,
    transaction_id VARCHAR(100) REFERENCES transactions(transaction_id),
    lock_id VARCHAR(50) REFERENCES devices(lock_id),
    action_type VARCHAR(20) NOT NULL, -- raise, lower, reset
    hex_command VARCHAR(100) NOT NULL,
    reason TEXT NOT NULL,
    operator_name VARCHAR(100) NOT NULL,
    executed_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

2. Time-Series / NoSQL Database (e.g. MongoDB or InfluxDB)

Digunakan untuk menyimpan log telemetri berkecepatan tinggi, delta perubahan sensor, dan data error mentah langsung dari perangkat parkir IoT.

Rekomendasi Tech Stack Modern B2B IoT

Arsitektur terdistribusi terintegrasi untuk menangani ribuan perangkat parkir model SC1240 secara skalabel:

IoT Gateway & Backend (API & Webhook)

  • Runtime Environment: Node.js (TypeScript) dengan framework NestJS. Menawarkan arsitektur modular tinggi, integrasi WebSocket/MQTT bawaan untuk telemetri IoT, dan pustaka ORM (TypeORM/Mongoose).
  • IoT Protokol: EMQX sebagai MQTT Broker berkinerja tinggi untuk mendengarkan telemetri perangkat via jaringan seluler (NB-IoT/4G).
  • Payment Integration: Node.js SDK Midtrans/Xendit untuk penanganan enkripsi, verifikasi signature webhook SHA-512, dan pembuatan QRISEMVCo dinamis.

Mobile Apps (Teknisi & Operator Lapangan)

  • Framework: React Native (dengan TypeScript). Menggunakan kode dasar tunggal (single codebase) untuk iOS & Android.
  • BLE Integrasi: Pustaka `react-native-ble-plx` untuk komunikasi nirkabel direct Bluetooth ke mikrokontroler SC1240 untuk keperluan OTA update, debug lokal, dan manual command reset lapangan.
  • State Management: Redux Toolkit dengan RTK Query untuk caching telemetry data dari API backend.

Infrastructure & Security

  • Cache & Queue: **Redis** untuk antrean retry pengiriman perintah BLE/NB-IoT (Retry Queue) ke perangkat yang sedang offline, serta menjaga ID idempotensi pembayaran agar tidak terjadi double payment.
  • Security: Enkripsi AES-128 pada frame komunikasi BLE, otentikasi JWT Bearer token pada API Backend, dan otorisasi bertingkat antara staf keamanan (Bypass) dan manager keuangan (Laporan Ekspor).

Contoh Query SQL Analitik Keuangan

Query relasional SQL di bawah digunakan oleh tim Finance dan Analyst untuk menghitung total pendapatan harian dari transaksi QRIS yang berstatus 'Paid' atau 'Settled' (dalam zona waktu Jakarta):

Query Pendapatan Harian

SELECT 
    DATE(entry_time AT TIME ZONE 'UTC' AT TIME ZONE 'Asia/Jakarta') AS tanggal_lokal,
    COUNT(transaction_id) AS jumlah_transaksi_sukses,
    SUM(amount_idr) AS total_pendapatan_bersih,
    SUM(CASE WHEN status = 'error' THEN amount_idr ELSE 0 END) AS potensi_kebocoran_status
FROM 
    transactions
WHERE 
    status IN ('paid', 'settled', 'completed')
    AND entry_time >= CURRENT_DATE - INTERVAL '30 days' -- Filter data 30 hari terakhir
GROUP BY 
    tanggal_lokal
ORDER BY 
    tanggal_lokal DESC;

Penjelasan Teknis Query:

  • `AT TIME ZONE` conversion: Mengonversi waktu penyimpanan standar UTC database ke waktu lokal operasional (`Asia/Jakarta`) agar kalkulasi pergantian hari tepat pukul 00:00 WIB.
  • `status IN ('paid', 'settled')`: Menyaring hanya transaksi yang telah terverifikasi sukses dibayar melalui webhook payment gateway.
  • `SUM` dan `COUNT`: Mengagregasikan total nominal rupiah dan menghitung volume transaksi per hari.
